# 车载以太网 & LIN 总线 — 面试速通手册

> 面试被问"你知道 CAN 之外还有哪些车载总线"时，能用 3 分钟讲清楚 LIN + 车载以太网的定位、区别和测试要点。

---

## 一、先看全景：三总线各司其职

```
         带宽
          ↑
  1 Gbps  │                              ● 车载以太网 (1000BASE-T1)
          │                               ADAS 摄像头 / 雷达融合
  100 Mbps│                              ● 车载以太网 (100BASE-T1)
          │                               OTA 升级 / 诊断 (DoIP)
    8 Mbps│              ● CAN FD
          │              动力 / 底盘实时控制
    1 Mbps│          ● CAN (经典)
          │          制动 / 转向 / 发动机
   20 Kbps│  ● LIN
          │   车窗 / 座椅 / 雨量传感器 / 空调风门
          └──────────────────────────────────────────→ 成本
              低 ←                                    → 高
```

一句话记住：**LIN 管身体（车身电子）、CAN 管手脚（动力底盘）、以太网管大脑（ADAS + 信息娱乐）**。

## 二、LIN 总线

### 一句话定义

LIN（Local Interconnect Network）是单线主从架构的低成本总线，用在对实时性要求不高、数据量小的车身电子节点上。

### 为什么要有 LIN

CAN 总线每个节点都需要晶体振荡器和 CAN 控制器，硬件成本高。车窗升降、座椅调节这些功能一秒钟传一帧就够了，用 CAN 是浪费。LIN 的从节点不需要晶振（用 RC 振荡器同步），硬件成本不到 CAN 的 1/3。

### 核心特征

| 维度 | LIN | CAN（对比参考） |
|---|---|---|
| 拓扑 | 单线（LIN + GND），菊花链 | 双绞线（CAN_H + CAN_L） |
| 速率 | 最高 20 kbit/s | 1 Mbit/s（经典） |
| 节点数 | 1 主 + 最多 16 从 | 理论无上限（ID 仲裁） |
| 通信模式 | 主从轮询（调度表驱动） | 多主事件触发（CSMA/CA） |
| 帧长度 | 2/4/8 字节数据 | 0-8 字节（经典）/ 0-64 字节（FD） |
| 错误检测 | 8 位校验和 | 15 位 CRC |
| 典型应用 | 雨量传感器、氛围灯、空调风门 | 发动机 ECU、ESP、EPS |

### 调度表（Schedule Table）— LIN 最核心的概念

CAN 是谁有数据谁就发（事件触发），LIN 是**主节点按时间表点名，点到谁谁才能发**。这张表叫 Schedule Table：

```
时间槽  0     10ms   20ms   30ms   40ms   50ms
       │      │      │      │      │      │
主节点： [ID=0x30] → [ID=0x31] → [ID=0x30] → [ID=0x32] → [ID=0x30] → ...
         │            │            │            │
从节点A： └── 回复 ──┘            └── 回复 ──┘
从节点B：              └── 回复 ──┘

ID=0x30: 车窗位置（周期性 20ms，高优先级）
ID=0x31: 雨量传感器（周期性 50ms，低优先级）
ID=0x32: 诊断请求（事件触发，sporadic slot）
```

关键点：
- 调度表是**确定性**的——每个时间槽发什么报文是预先设计好的
- 实际项目里 LIN 调度表通常由 Vector LDF（LIN Description File）文件定义
- 面试被问"LIN 怎么保证实时性"——答：不靠仲裁，靠**确定性调度**，时间槽是固定分配的

### LIN 帧格式（看一眼就行）

```
┌──────────┬──────────┬────────┬────────┬──────────┬──────────┐
│ Sync     │ Sync     │ PID    │ Data   │ Data     │ Checksum │
│ Break    │ Field    │        │ 0      │ ...7     │          │
│ ≥13 bit  │ 0x55     │ 6bit+2 │        │ 0-8 byte │ 8 bit    │
└──────────┴──────────┴────────┴────────┴──────────┴──────────┘
```

帧头由主节点发出，数据由从节点回复。PID（Protected ID）的低 6 位就是 CAN 里你熟悉的那种 ID，高 2 位是奇偶校验。

### 面试这么说 LIN

> "LIN 用在车身电子，单线主从架构，最高 20kbps。核心机制是调度表——主节点按时隙依次发送帧头，从节点回复数据，通信是完全确定性的。我在台架上如果要测试 LIN 节点，最关键的是要能模拟主节点发送正确的调度表，然后用 LIN 工具（如 Vector VN1610）抓取从节点的回复帧做校验。"

## 三、车载以太网

### 一句话定义

车载以太网是基于 IEEE 802.3bp（100BASE-T1）/ 802.3ch（10GBASE-T1）标准的高速车载网络，用单对非屏蔽双绞线实现全双工通信，用于 ADAS 传感器数据传输、OTA 升级和整车诊断。

### 为什么要有车载以太网

传统 CAN 瓶颈在 1 Mbps，CAN FD 也只能到 8 Mbps。一个高清 ADAS 摄像头的原始数据流大约 1 Gbps，CAN 完全扛不住。车载以太网 100 Mbps 起步（100BASE-T1），10 Gbps 的多千兆以太网也在路上了。另一个驱动力是架构变革——整车从**域集中式**（每个域一个网关）向**中央计算 + 区域控制**（Zonal Architecture）演进，只有以太网的交换式拓扑能支撑这种架构。

### 车载以太网 vs 标准以太网

| 维度 | 车载以太网 (100BASE-T1) | 标准以太网 (100BASE-TX) |
|---|---|---|
| 物理介质 | 1 对非屏蔽双绞线 | 2 对（4 根线，CAT5） |
| 全双工 | 是（同对线同时收发） | 是（收和发各用一对） |
| 编码 | PAM-3 | MLT-3 |
| 抗干扰 | 强，专为汽车 EMI 设计 | 弱，需要 STP 屏蔽 |
| 重量/成本 | 显著更低 | 高 |

核心物理层差异：100BASE-T1 在一对线上同时收发（回声消除 + 混合电路），而不是像标准以太网那样收发各用一对线。这就是为什么线束能省一半。

### 三个面试必知的上层协议

**SOME/IP（Scalable service-Oriented MiddlewarE over IP）**

CAN 是信号导向的（Signal-based）——每个 CAN ID 对应固定的一组信号。"车速"永远在 0x0C0 的第 0-1 字节。车载以太网是**服务导向的（Service-based）**——"谁有车速信息？"通过 SOME/IP 的服务发现（Service Discovery）动态查找。这是你能在 ADAS 域控制器和摄像头之间灵活传输数据的核心协议。

**DoIP（Diagnostics over IP）**

把 UDS 诊断协议从 CAN 物理层搬到了以太网上。作用是：刷写 ECU 固件时不用再等 CAN 的 1 Mbps 慢慢传——100 Mbps 的 DoIP 可以把刷写时间从 30 分钟缩到 2 分钟。台架测试里，批量 ECU 的固件更新几乎全靠 DoIP。

**TSN（Time-Sensitive Networking）**

以太网天生是尽力而为（best-effort）的，但制动控制需要确定性延迟。TSN 是 IEEE 802.1 工作组对以太网做的"实时改造"——通过时间同步（802.1AS）和流量整形（802.1Qbv），让关键帧在精确的时间窗口内送达，延迟抖动降到微秒级。

### 面试这么说车载以太网

> "车载以太网主要解决 CAN 的带宽瓶颈和架构演进需求。物理层上 100BASE-T1 在单对双绞线上实现 100 Mbps 全双工，上层用 SOME/IP 做服务化通信、DoIP 做高速诊断刷写、TSN 做时间敏感流量的确定性传输。台架测试里如果要注入以太网传感器数据（比如通过以太网给 ADAS 域控送仿真的摄像头原始数据流），需要对 SOME/IP 的服务发现和数据序列化流程比较熟悉。"

## 四、台架测试视角：三者怎么测

| 测试项 | CAN | LIN | 车载以太网 |
|---|---|---|---|
| 总线监控 | Vector VN1640 | VN1610（带 LIN 收发器） | VN5640 / VN5650 |
| 残余总线仿真 | CANoe 模拟未接 ECU 的报文 | CANoe 模拟主节点调度表 | CANoe 模拟 SOME/IP 服务端 |
| 故障注入 | 短路/开路/CAN_H 对 CAN_L 短路 | 对地短路/对电源短路 | 链路断开/CRC 错误帧注入 |
| 数据记录 | 所有 CAN 帧 | 所有 LIN 帧（含调度表） | 所有以太网帧（PCAP 格式） |
| 刷写测试 | 通过 CAN 做 Bootloader | 通过 LIN 做诊断 | 通过 DoIP 高速刷写 |
